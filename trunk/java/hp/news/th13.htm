Um thread pode "dormir"

Diz-se que um thread encontra-se "dormindo" quando este executa o método sleep(). Há duas variantes deste método: (a) uma que recebe um valor em milissegundos e outra que recebe, além do valor em milissegundos, um valor adicional em nanossegundos. Em ambos os casos, a informação fornecida estabele quanto tempo o thread em questão irá permanecer no estado bloqueado.

O thread permanece no estado bloqueado até que seja expirado o intervalo de tempo fornecido ao método sleep(). Quando este intervalo de tempo é transcorrido, o thread passa para o estado executável. (Vou repetir, bem devagar e em bom tom. Sugiro que também repita, em voz baixa!) O thread sai do estado bloquado para o estado executável. Ou seja, retorna para o conjunto de threads que ansiosamente aguardam para serem executados. Não retorna, portanto, ao estado executando.

Conclui-se daí que o método sleep() não é apropriado para estabelecer uma interrupção exata de tempo. A interpretação mais exata é: o método sleep() faz com que o thread permaneça no estado bloqueado pelo menos o intervalo de tempo fornecido como argumento. O retorno à execução, contudo, não é imediato, visto que após transcorrido este intervalo de tempo, o thread vai para o estado executável, e permanece lá até que seja selecionado para execução.
